Evaluating SQL vs. NoSQL for semi-structured data

WhyNoSQL Project: Big Data. Semi-structured data. Experimenting. Performance metrics. Databases. NoSQL.

Context and Background

Relational vs NoSQL Databases

Relational or Structured Query Language (SQL) databases have long been the preferred and default choice for new database management system (DBMS) deployments. But for certain types of data they are not always most efficient. Typically NoSQL databases are faster than relational databases, but this dependant on dataset and implementation.

Big Data and Semi-Structured Data

With the rise of Big Data and Web 2.0 applications, there are different types of data with various structures that need to be operated on whilst observing the three V’s of Big Data. We are focussed on the performance of databases on semi-structured data and datasets that comprise sparse and varied elements.

Validating Performance Claims

To validate the claims of NoSQL with clustering, we implemented query operations, two NoSQL databases, MongoDB and Cassandra, and a relational database, MySQL, in various cluster configurations and evaluated performance for each DBMS design and configuration to capture elapsed metrics.​

Project Goals

Our project goals were to implement three database designs. Two of these designs were NoSQL implementations and additionally a relational database was implements for comparison. These designs supported Twitter datasets which comprised of a semi-structured data with multiple missing elements and varying document lengths. Our intention was to then test performance for each of these database designs in various configurations and schema definitions to document the most performant design for our semi-structured dataset, one that is both generic and representative of similar big data scenarios.

Experiments

We conducted experimentation in a uniform computing environment where each node used identical commodity hardware and was only hosting the database instance itself. We designed tests to benchmark the read and write performance of the database candidates for each cluster design or configuration.

Five query operations to be executed on the databases were implemented:

  • Bulk-insertions - loading a whole dataset
  • Single-insertions - single record insert
  • Indexed-inserts - single record insert into indexed data model
  • indexed-selects - retrieving data from indexed data model
  • Scanning - scanning whole database
